# Prompt: 文档基线重构（baseline-doc-refactor）

> 用途：对一个 AI 上下文文档已经腐化（流水账、冲突、过期）的项目执行一次性基线重构。
> 本提示词是实践验证过的重构任务的通用化版本。

------

## 任务

对项目 `{{PROJECT_PATH}}` 的 AI 上下文文档执行基线重构，建立"先探索事实、再自主维护、旧结论原位替换、历史内容归档"的长期机制。

## 核心治理原则

1. 先探索项目事实，再修改文档；当前代码/配置/迁移/测试优先于汇总文档。
2. 旧结论失效时原位替换或删除；禁止追加"最新补充"或相反结论。
3. 同一事实在活动上下文中只保留一份当前结论。
4. 历史过程有追溯价值时移入 archive/，头部注明"冻结快照/非当前事实"。
5. 05_PROJECT_CONTEXT.md 只保存当前有效上下文，不做里程碑流水账。
6. 不能因为担心误删而保留所有历史内容；也不得误删仍有效的生产/权限/安全约束。

## 步骤

1. 完整审计入口文件与 docs/ai 活动文件：职责、重复、冲突、失效、流水账。
2. 建立当前事实清单：按事实来源优先级（运行证据 > 代码 > 迁移 > 配置 > 测试 > 用户确认 > 专题文档 > 汇总 > 历史）核验组件/环境/数据库/鉴权/隔离/门禁/阶段状态。
3. 对 05 做 KEEP / REWRITE / MERGE / DELETE / ARCHIVE 分类。
4. 归档历史内容并更新 README 导航（归档放"历史追溯区"，不进常用入口）。
5. 重写 05 为"当前事实文档"（参考 project-templates/docs-ai/05 模板的十四节结构）。
6. 在入口文件与治理规则中接入"AI 文档自治维护"机制（见 core/05_DOCUMENT_GOVERNANCE_RULES.md）。
7. 全文一致性检查：状态冲突扫描、双入口一致性、引用路径有效性、`git diff --check`。
8. 独立 docs commit + 完整报告（探索范围/重构结果/归档结果/治理机制/一致性验证/未修改内容）。

## 禁止

- 只在文末追加治理章节；保留旧错误结论注明"以新内容为准"；
- 默认遍历全部专题文档；根据未验证计划重写事实；
- 与业务代码混提交；把本轮过程写回 05。
