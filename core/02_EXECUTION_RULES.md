# AI Coding Execution Rules

> 本文档是 AI Coding Agent 的执行规范（AICoding-RULE 通用基线）。
>
> 在完成阅读规范后，所有代码修改必须遵守本规范。
>
> 本规范优先级低于：
>
> ```text
> 01_READING_RULES.md
> 05_PROJECT_CONTEXT.md
> ```
>
> 但高于测试规范和输出规范。

------

# 1. 核心原则

执行阶段目标：

```text
正确
稳定
可验证
可回滚
```

而不是：

```text
优雅
复杂
未来可能有用
```

原则：

```text
最小修改优先

一致性优先

可维护优先

可验证优先
```

------

# 1A. 阶段最终目标与边界总控

任务按阶段推进时，必须遵守阶段总控。

## 1A.1 阶段定义要求

每个阶段必须有明确的：

```text
最终目标

允许范围

禁止事项

验收标准
```

## 1A.2 阶段开始前必须复述

```text
本阶段目标：

本阶段允许修改：

本阶段禁止触碰：

本阶段验收标准：
```

## 1A.3 阶段执行边界

```text
不得把多个阶段混在同一轮完成

不得提前实现后续阶段能力

不得用"顺便完成了某功能"替代阶段验收
```

## 1A.4 阶段结束后必须输出

```text
是否达成阶段目标

是否越界

是否遗留问题

下一阶段入口
```

------

# 1B. 懒惰开发阶梯

写任何代码前，停在第一个成立的台阶上：

```text
1. 这件事需要做吗？（YAGNI）

2. 项目里已经有现成实现吗？复用已有 helper / util / 模式。

3. 标准库能做吗？用标准库。

4. 平台原生能力能覆盖吗？用原生能力。

5. 已安装的依赖能解决吗？用已有依赖。

6. 能一行解决吗？写一行。

7. 以上都不行，才写最少能工作的代码。
```

阶梯在理解问题之后运行，不能替代理解问题。

有意的简化必须留注释标记：说明简化的上限（全局锁、O(n²) 扫描、朴素启发式等）和升级路径。

懒惰的边界——以下内容不允许偷懒：

```text
理解问题（完整读需求、走通真实链路）

信任边界上的输入校验

防止数据丢失的错误处理

安全

无障碍

显式要求的内容
```

非平凡逻辑必须留下一个可运行的最小检查（断言式自检或一个小测试文件）；平凡一行代码不需要。

------

# 2. 最小修改原则

只修改当前任务直接相关内容。

禁止：

```text
修复 Bug 时顺手重构

新增功能时顺手优化架构

修改接口时顺手调整命名

修复样式时顺手重写页面
```

要求：

```text
只修改必要文件

只修改必要逻辑

保持现有行为兼容
```

------

# 3. 禁止提前抽象

AI 最容易犯的错误：

```text
为未来设计

为可能出现的需求设计

为不存在的扩展设计
```

禁止：

```text
BaseXXX

AbstractXXX

PluginSystem

GenericPipeline

StrategyFactory

EventBus

Registry
```

除非已有多个真实使用场景。

------

## 抽象升级规则

```text
第 1 个使用者
直接实现

第 2 个使用者
提取公共函数

第 3 个使用者
考虑提取公共组件

第 4 个使用者
考虑抽象层
```

允许少量重复代码存在。

------

# 4. 默认禁止新增依赖

新增依赖前必须说明：

```text
为什么需要？

现有依赖为什么无法满足？

是否有标准库替代？

是否影响部署？

是否影响构建？

是否影响安全？
```

未完成说明：

```text
禁止新增依赖
```

------

# 5. 文件规模控制

建议：

```text
单文件 ≤ 500 行

单函数 ≤ 80 行
```

超过后考虑拆分。

拆分原则：

```text
按职责拆分

按模块拆分

按功能拆分
```

禁止：

```text
为了拆分而拆分

制造大量空目录

制造大量中间层
```

项目上下文明确豁免时（例如"保持现有单体结构，不做无需求拆分"），以项目上下文为准。

------

# 6. 保持现有架构

优先遵循项目已有结构。

例如：

```text
API
↓
Service
↓
Repository
↓
Database
```

或者：

```text
Controller
↓
Domain
↓
Persistence
```

禁止：

```text
新增架构层

替换已有架构模式

改变项目主要组织方式
```

除非任务明确要求。

------

# 7. 数据模型规则

原则：

```text
数据结构清晰

关系明确

避免动态结构
```

禁止：

```text
JSON 套 JSON

无约束动态字段

一个实体承担多个职责
```

新增持久化字段：

```text
必须遵循项目现有迁移机制
```

禁止绕过项目数据库管理方式。

------

# 8. 状态管理规则

所有状态变化必须可追踪。

禁止：

```text
任意修改状态字段

绕过状态流转逻辑

跳过业务校验直接改状态
```

要求：

```text
通过项目已有状态管理机制修改状态

保持状态流转可验证

保证状态变化可回溯
```

------

# 9. 异常处理规则

禁止：

```python
except:
    pass
```

禁止：

```python
except Exception:
    return "失败"
```

要求：

```text
记录日志

保留上下文

提供有意义错误信息

区分业务异常和系统异常
```

------

# 10. 外部服务调用规则

涉及：

```text
LLM

邮件

支付

第三方 API

搜索服务

对象存储
```

要求：

```text
超时控制

异常处理

降级方案

日志记录
```

禁止：

```text
直接将底层异常暴露给用户
```

------

# 11. Prompt 与 LLM 规则

涉及 AI 功能时：

要求：

```text
Prompt 版本化

Prompt 集中管理

输出结构化

结果可校验

失败可恢复
```

禁止：

```text
大量 Prompt 写死在代码中

依赖自由文本解析

无约束输出格式
```

------

# 12. 数据库变更规则

涉及数据库结构修改：

必须：

```text
使用项目既有迁移机制

提供升级方案

提供回滚方案

分析历史数据兼容性
```

禁止：

```text
直接修改生产结构

绕过迁移机制

破坏历史数据兼容性
```

------

# 13. 高风险变更规则

以下属于高风险：

```text
数据库

认证

权限

配置

环境变量

部署

缓存

异步任务

文件存储

支付

第三方服务

LLM Prompt 与真实模型调用边界
```

涉及以上内容：

必须先完成：

```text
现状分析

风险分析

修改方案

验证方案

回滚方案
```

未完成分析：

```text
禁止修改
```

------

# 14. 验证优先原则

任何修改必须具备验证方式。

至少回答：

```text
如何证明修改成功？

如何证明旧功能未损坏？

如何复现问题？

如何验证修复？
```

无法验证：

```text
禁止提交
```

------

# 15. 执行完成后必须输出

必须输出：

```text
# 修改内容

# 修改文件

# 修改原因

# 风险等级

# 影响范围

# 验证方案

# 回滚方案
```

禁止只输出：

```text
已经修改完成
```

------

# 16. 最终原则

AI 的职责不是创造新的系统。

AI 的职责是：

```text
理解已有系统

遵循已有系统

安全修改已有系统
```

任何时候：

```text
阅读规范
>
项目上下文
>
执行规范
>
测试规范
>
输出规范
```

优先级不得颠倒。

------

# 17. BUG 修复前置探索原则

Bug 报告描述的是症状，不是根因。

修复前必须先澄清：

```text
真实调用链是什么？

真实根因在哪一层？

涉及哪些文件？

修改范围是什么？

是否影响无关模块？

是否可能引入回归？
```

必须排查的常见假根因：

```text
是否是缓存残留？

是否是旧构建产物 / 旧打包文件在运行？

是否是旧进程未重启，跑的不是最新代码？

是否是环境变量 / 配置覆盖导致？

是否只是数据问题而不是代码问题？
```

修复共享逻辑时，必须搜索该函数的全部调用方：

```text
优先在共享逻辑处修一次

禁止只修当前报告路径、留下旁路问题
```

禁止仅凭现象直接编写修复方案。

------

# 18. 修复计划审查前置条件

高风险区域的 Bug 修复（见 #13），修复计划必须先输出并经用户审查后才能执行。

修复计划至少包含：

```text
根因结论与证据

修改文件与范围

不修改内容

回归风险

验证方案
```

低风险局部修复可直接执行，但仍须遵守 #17 前置探索。

------

# 19. 高风险代码日志原则

高风险逻辑必须写诊断日志。

日志至少包含：

```text
stage：当前执行到哪一步

输入摘要：关键输入的脱敏摘要

判断结果：每个门禁 / 分支的判定值

failure_stage：失败发生在哪一步
```

禁止：

```text
只写"失败了"

只写"出错"

吞掉异常不留痕
```

脱敏要求：

```text
token、密钥、密码、手机号、
身份标识、原始请求 body 等
敏感信息必须脱敏后才能进日志
```

具体哪些逻辑属于"必须写诊断日志的高风险场景"，由项目在项目上下文中登记。

------

# 20. 产物与密钥禁提交清单

默认禁止提交：

```text
运行产物目录（runs/、output/、dist/ 等）

生成的媒体文件（视频、音频、大图）

.env 及任何真实环境配置

API key、token、证书、密钥

LLM raw response 落盘文件

本地数据库文件
```

项目可在项目上下文中追加或收紧本清单，不得放宽。

------

# 21. 诊断 / 调试接口安全规则

诊断、调试类接口（debug、inspect、dump 等）：

```text
不得返回原始运行时对象
（UI 控件、连接对象、ORM 对象、循环引用结构）

返回值必须经过安全序列化

内部异常应返回结构化错误（success=false + 原因），
而不是裸 500 / 堆栈泄漏

不得泄漏敏感配置与密钥
```
