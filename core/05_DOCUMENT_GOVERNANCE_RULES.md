# AI Document Governance Rules

> 本文档是 AI 协作文档的治理规范（AICoding-RULE 通用基线）。
>
> 适用对象：项目入口文件（CLAUDE.md / AGENTS.md）、docs/ai 活动文件
>（README、01~04 规则、05 项目上下文）及其归档。
>
> 本规范与 01~04 同级加载；治理层修改门槛见 #8。

------

# 1. 文件职责分层

```text
CLAUDE.md / AGENTS.md   入口协议 + 项目硬约束摘要（等效双入口，必须语义一致）

01_READING_RULES.md     怎么读项目（稳定规则）

02_EXECUTION_RULES.md   怎么改代码（稳定规则）

03_TESTING_RULES.md     怎么验证（稳定规则）

04_OUTPUT_RULES.md      怎么汇报（稳定规则）

05_PROJECT_CONTEXT.md   当前项目事实文档（唯一随项目状态变化的活动文件）

docs/ai/README.md       按需导航（不要求遍历全部专题文档）

专题目录 / archive/      详情与历史（默认不读，按需进入）
```

核心分层原则：

```text
规则层负责"怎么工作"，与项目状态无关。

事实层（05 + 专题）负责"项目现在是什么样"。

项目事实禁止写入规则层正文；
规则层只允许留占位机制（"由项目在 05 登记"）。
```

------

# 2. 维护责任

```text
AI 文档由 AI Coding Agent 自主维护。

开发者只负责：
  审阅 Git diff
  裁决高风险业务结论
```

发现文档与事实不符时，AI 必须主动修复，不得等待开发者指示。

------

# 3. 先探索后更新

```text
禁止仅根据对话摘要、旧任务报告或现有汇总文档判断当前事实。

更新任何结论前，必须先定位对应的
代码 / 配置 / 迁移 / 测试 / 运行证据。
```

事实来源优先级（冲突时从高到低）：

```text
1. 已验证的实际运行证据
2. 当前有效代码
3. 数据库迁移和 schema
4. 当前配置、环境模板和 Compose
5. 自动化测试及验收脚本
6. 用户已经确认的业务决定
7. 当前专题文档
8. AI 汇总上下文
9. 历史报告和归档文件
```

------

# 4. 每轮文档影响检查

每轮任务和每个大阶段完成后，必须执行一次文档影响检查：

```text
本轮改动是否使以下内容过期？

  05_PROJECT_CONTEXT.md 中的当前事实
  入口文件中的硬约束
  README 导航
  相关专题文档
```

受影响 → 同轮更新；不受影响 → 在完成报告中明确说明"无文档影响"。

------

# 5. 原位替换规则

```text
旧结论失效时：原位替换或删除。

禁止：
  在错误结论后追加"最新补充""当前更新"或相反结论
  保留旧错误结论并注明"以新内容为准"
  同一事实在活动上下文中保留多份结论
```

------

# 6. 自主更新触发条件

出现以下任一情况必须更新文档：

```text
1. 服务、端口、部署入口、环境边界变化
2. 数据库、迁移、切换状态变化
3. 鉴权、权限、隔离边界变化
4. 安全门禁口径变化
5. 阶段状态变化（完成 / 取消 / 延期 / 阻塞）
6. 用户确认了新的业务决定
7. 发现活动文档中存在事实错误或内部冲突
```

升级为长期规则的条件：

```text
同类问题重复出现，或严重风险首次出现
→ 写入对应规则文件（走 #8 门槛）
```

不应写入活动文档的内容：

```text
1. 未验证的计划、猜测、临时调试结论
2. 单次任务的详细过程（写任务报告，不写进上下文）
3. 提交号、测试通过数量、修改文件清单等流水账
   （仅在定位关键边界确有必要时保留提交号）
4. 样例级微调结论（针对单个样本 / 单次演示的参数，
   不得升格为全局规则）
```

------

# 7. 防膨胀规则

```text
05_PROJECT_CONTEXT.md 只保存当前有效上下文：

  不按日期追加里程碑流水账
  不复制专题文档的大段内容
  同一事实只保留一份当前结论
  建议控制在数十 KB 量级；
  接近或超过 80 KB 时必须触发一次瘦身审计

规则文件本身同样防膨胀：

  项目专属细节进 05 或项目扩展区，不进规则正文
  过期的任务期章节必须删除或归档，不得沉积
```

------

# 8. 治理规则文件的较高修改门槛

```text
01~05 规则文件（含本文件）属于治理层。

修改治理层需要满足：
  问题重复出现或风险等级高
  修改后不与现有规则冲突
  在任务报告中说明修改理由

日常事实更新只改 05_PROJECT_CONTEXT.md 和专题文档。

修改入口文件硬约束时，
CLAUDE.md 与 AGENTS.md 必须同步，保持语义一致。
```

使用受控区块（managed block）的项目：

```text
区块内为通用基线注入内容，禁止手工修改；
升级通过安装脚本进行。
项目扩展写在区块外的扩展区。
```

------

# 9. 归档原则

```text
进入归档（docs/ai/archive/ 或专题目录）：
  已完成阶段的详细调查过程
  废弃架构和旧迁移方案
  历史运行证据和长测试报告
  仅用于解释"为什么曾经这样做"的内容

活动上下文最多保留一句仍然必要的历史提醒。
```

归档文件必须：

```text
头部注明"冻结快照 / 历史追溯用，非当前事实"

可从 README 或专题入口找到

不再维护"当前状态"

不进入"当前常用入口"，
单独放在"历史追溯 / 归档入口"区，
并注明：普通任务不得默认读取
```

推荐归档目录形态（按项目规模二选一）：

```text
小型项目：archive/{explore, design, implementation, acceptance}

大型项目：按业务域专题目录 + archive/ 冻结快照区
```

------

# 10. 交接快照（HANDOFF）机制（可选）

多人 / 多窗口交接频繁的项目，可维护 docs/ai/HANDOFF.md：

```text
内容：当前结论、当前流程、硬规则、验收方式、后续注意

阅读顺序：先 HANDOFF，再 05_PROJECT_CONTEXT.md

约束：
  HANDOFF 是快照不是流水账，每次交接原位重写
  样例级微调必须显式标注，不得写成全局规则
```

------

# 11. 修改后一致性检查

每次修改活动文档后必须自查：

```text
1. 同一事实是否只剩一份结论
2. 是否引入了"已完成"与"进行中"并存
3. CLAUDE.md 与 AGENTS.md 硬约束是否语义一致
4. README 导航是否仍然有效（引用路径存在）
5. 是否误删了仍有效的生产、权限、数据、安全约束
```

------

# 12. 文档维护任务的报告要求

涉及文档维护的任务，报告中必须包含：

```text
修改了哪些文档、为什么
替换 / 删除了哪些旧结论
归档了什么
执行了哪些一致性检查
明确说明"无文档影响"的部分
```
