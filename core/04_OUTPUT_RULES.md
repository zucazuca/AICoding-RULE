# AI Communication & Output Protocol

> 本文档定义 AI Coding Agent 与开发者之间的沟通协议（AICoding-RULE 通用基线）。
>
> 本规范用于约束：
>
> - 如何汇报阅读结果
> - 如何汇报方案
> - 如何汇报实现结果
> - 如何汇报测试结果
> - 如何说明风险
> - 如何请求确认
>
> 本规范优先级低于：
>
> ```text
> 01_READING_RULES.md
> 05_PROJECT_CONTEXT.md
> 02_EXECUTION_RULES.md
> 03_TESTING_RULES.md
> ```
>
> 但高于默认模型行为。

------

# 1. 核心原则

AI 的职责不仅是修改代码。

还必须：

```text
解释自己做了什么

解释为什么这样做

解释风险在哪里

解释如何验证
```

禁止：

```text
直接修改

直接结束

不说明原因
```

------

# 2. 事实与推测分离

所有分析必须区分：

```text
事实（Fact）

推测（Hypothesis）
```

禁止：

```text
我觉得

我猜

应该

可能
```

作为事实输出。

正确格式：

```text
# Fact

在 user_service.py 中发现权限校验逻辑。

# Hypothesis

缓存层可能导致结果未及时更新，需要进一步验证。
```

------

# 3. 证据优先原则

所有结论必须尽可能提供依据。

优先引用：

```text
文件路径

类名

函数名

接口名

配置项

日志内容

测试结果
```

禁止：

```text
项目应该是这样设计的

这里看起来没问题

大概率不会影响其他功能
```

没有证据支撑的判断。

------

# 4. 风险等级标准

所有修改必须标记风险等级。

```text
LOW

MEDIUM

HIGH
```

定义：

### LOW

```text
局部逻辑修改

文档修改

测试补充

样式调整

无状态修改
```

### MEDIUM

```text
跨模块修改

新增接口

新增业务逻辑

状态流转调整

缓存逻辑修改
```

### HIGH

```text
数据库

权限

认证

配置

部署

第三方服务

支付

文件存储

LLM Prompt 与真实模型调用边界
```

------

# 5. 确认机制

AI 必须明确说明：

```text
是否需要用户确认
```

格式：

```text
需要确认：YES

或

需要确认：NO
```

------

## 必须确认的场景

```text
数据库结构变更

接口协议变更

权限逻辑变更

认证逻辑变更

配置项变更

部署变更

删除代码

删除数据

删除文件

修改 Prompt / 默认启用真实 LLM 调用（如项目涉及）
```

------

## 可直接执行的场景

```text
局部 Bug 修复

单元测试补充

文档修正

日志补充

低风险重构
```

------

# 5A. 阶段目标总控输出格式

## 阶段开始前必须输出

```text
# 阶段目标复述

本阶段目标：

允许范围：

禁止事项：

验收标准：
```

## 阶段结束后必须输出

```text
# 阶段收口检查

是否达成目标：

是否越界：

是否提前实现后续阶段能力：

遗留问题：
```

------

# 6. 阅读阶段输出格式

完成阅读后：

```text
# 阅读结果

## 项目理解

## 当前模块

## 当前调用链

## 已阅读文件

## 业务语义确认

## 风险等级

## 下一步建议
```

禁止：

```text
我已经理解项目

问题不大

可以开始改
```

------

# 7. 方案阶段输出格式

完成方案分析后：

```text
# 方案设计

## 问题分析

## 根因分析

## 修改方案

## 不修改内容

## 影响范围

## 风险分析

## 验证方案

## 是否需要确认
```

------

# 8. 实现阶段输出格式

完成代码修改后：

```text
# 实现结果

## 修改文件

## 修改内容

## 修改原因

## 影响范围

## 风险等级

## 验证方式

## 是否需要确认
```

------

## 修改文件格式

推荐：

```text
src/services/user_service.py
- 修复权限判断逻辑

src/api/user.py
- 调整接口参数校验
```

禁止：

```text
修改了一些文件
```

------

# 9. 测试阶段输出格式

完成测试后：

```text
# 测试结果

## 已执行测试

## 未执行测试

## 测试结论

## 剩余风险

## 建议人工验证
```

------

## 已执行测试

必须明确：

```text
执行了什么

结果是什么
```

例如：

```text
Unit Test
PASS

Integration Test
PASS

Manual Verification
未执行
```

------

# 10. 未执行测试说明

如果测试未执行：

必须说明原因。

例如：

```text
未执行 E2E

原因：
当前环境缺少数据库连接
```

禁止：

```text
测试通过
```

但实际没有测试。

------

# 11. 剩余风险说明

任何修改完成后必须说明：

```text
当前仍然存在什么风险
```

例如：

```text
缓存相关逻辑未验证

生产配置未验证

高并发场景未验证
```

禁止隐藏风险。

------

# 12. 下一步建议

每次回复结尾必须包含：

```text
# 下一步建议
```

可选：

```text
继续阅读

进入方案设计

等待确认

开始实现

执行测试

部署验证
```

禁止直接结束回复。

------

# 13. 不确定时的行为

如果证据不足：

必须明确说明：

```text
信息不足

需要进一步阅读

需要查看更多文件

需要用户补充信息
```

禁止：

```text
直接猜测

直接编造调用链

直接假设业务逻辑
```

------

# 14. 禁止行为

禁止：

```text
隐藏风险

省略影响范围

跳过验证说明

未测试声称已测试

将推测当作事实

擅自决定高风险变更

未确认删除数据

未确认删除文件

未确认修改数据库结构
```

------

# 15. 最终原则

AI 不仅要给出答案。

更要让开发者知道：

```text
为什么得到这个答案

依据是什么

风险是什么

下一步做什么
```

任何时候：

```text
可验证

可解释

可追溯

优先于简短输出
```

------

# 16. 任务完成报告规范

任务完成报告必须包含以下全部项：

```text
1. 真实根因（如为修复类任务）

2. 修改文件

3. 是否触碰项目登记的敏感模块 / 高风险区域

4. 是否有安全风险

5. 是否影响旧链路

6. 测试结果

7. 真机 / 真实环境验证结果（如适用）

8. 未解决问题

9. 不提交文件

10. commit 建议

11. 文档影响检查结论
    （受影响文档已更新 / 明确"无文档影响"，
    详见 05_DOCUMENT_GOVERNANCE_RULES.md）
```

禁止只输出"修改完成"而不包含以上内容。

项目可在项目层追加报告项（例如"是否调用真实 LLM""是否生成媒体产物"），不得删减基线项。
